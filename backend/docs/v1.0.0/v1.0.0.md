# Olmeware Store — Backend API v1.0.0

Go backend for the Olmeware storefront. Serves catalog, cart, orders, and
payments; the frontend only renders what this API composes.

- **Base URL (dev):** `http://localhost:8000`
- **API prefix:** `/api/v1` (every endpoint is versioned)
- **Auth:** JWT bearer access tokens (15 min) + opaque rotating refresh tokens (30 days)
- **Money:** integer **minor units** (MXN centavos). `44900` = `$449 MXN`. Responses
  also include a preformatted `price` string.
- **Errors:** uniform envelope `{"error":{"code","message"}}` with the matching HTTP status.
- **Logging:** every request is logged to stdout (method, path, status, duration, ip, request id).

## Stack

Go 1.26 · `net/http` (stdlib router, method+path patterns) · pgx/v5 (Postgres/Supabase) ·
golang-jwt/v5 · x/crypto/bcrypt · stripe-go/v79 · Coinbase Commerce (crypto). No obscure
dependencies.

## Runtime

```bash
# from backend/
go run . -seed     # idempotently seed admin + catalog, then exit
go run .           # start the API on :8000
go build -o bin/olmeware . && ./bin/olmeware
```

Environment (`.env.local`): `DATABASE_URL`, `JWT_SECRET`, `FRONTEND_URL`,
`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `COINBASE_COMMERCE_KEY`,
`COINBASE_COMMERCE_WEBHOOK_SECRET`, `ADMIN_EMAIL`, `ADMIN_PASSWORD`, `PORT` (default 8000).

## Roles

A normal store: everyone is a `customer`. Exactly one bootstrap `admin`
(`ADMIN_EMAIL`, default `admin@olmeware.store`) manages catalog. Admin-only routes require
an admin bearer token.

## Auth — `/api/v1/auth`

| Method | Path | Auth | Body | Description |
| --- | --- | --- | --- | --- |
| POST | `/auth/register` | – | `{name,email,password}` | Create a customer; returns a session. `409` on duplicate email. |
| POST | `/auth/login` | – | `{email,password}` | Authenticate; returns a session. `401` on bad credentials. |
| POST | `/auth/refresh` | – | `{refreshToken}` | Rotate tokens; old refresh token is revoked. |
| POST | `/auth/logout` | – | `{refreshToken}` | Revoke the refresh token. |
| GET | `/auth/me` | Bearer | – | Current user. |

**Session response**
```json
{
  "user": {"id","email","name","role","status","createdAt"},
  "accessToken": "<jwt>",
  "refreshToken": "<opaque>",
  "expiresAt": 1784707779
}
```
Passwords are bcrypt-hashed (cost 12) and never returned. Refresh tokens are stored only
as SHA-256 hashes in `user_sessions`.

## Catalog (public, backend-composed) — `/api/v1/catalog`

### `GET /catalog/products`
Storefront listing. Query params:

| Param | Example | Notes |
| --- | --- | --- |
| `garment` | `hoodie` | shirt \| sweater \| hoodie \| cap |
| `stack` | `frontend` | languages, frontend, backend, ai-ml, devops, tools |
| `size` | `L` | products having an active variant in this size |
| `collection` | `new-arrivals` | collection slug |
| `search` | `react` | matches name or tech label |
| `featured` | `true` | featured only |
| `minPrice` / `maxPrice` | `400` / `900` | **major** MXN units |
| `sort` | `price_desc` | `newest` (default), `price_asc`, `price_desc`, `name` |
| `limit` / `offset` | `60` / `0` | pagination (limit ≤ 100) |

Response: `{"products":[Product],"count":n}`.

**Product**
```json
{
  "id","slug","name","description","garment","stack","tech",
  "logo": "/logos/react.svg",
  "colorHex": "#1a1a1a",
  "priceMinor": 89900, "price": "$899 MXN", "currency": "MXN",
  "featured": true, "status": "active",
  "sizes": ["S","M","L","XL","XXL"],
  "images": [],
  "collections": ["new-arrivals"]
}
```

### `GET /catalog/products/{slug}`
Single product + purchasable variants with live availability. `404` if not found. Adds:
```json
"variants": [
  {"id","sku","size","colorHex","colorName","priceMinor","inStock":true,"available":100}
]
```

### `GET /catalog/collections`
`{"collections":[{"id","slug","name","description","productCount"}]}` (active only).

### `GET /catalog/tech-themes?category=`
Logo library. `{"themes":[{"id","slug","name","category","logo"}]}`. Logo URLs are stored
in the DB and served to the frontend.

## Cart — `/api/v1/cart`

Works for guests and authenticated users. A guest cart is addressed by an opaque
**`X-Guest-Token`** header; the first mutating call by a tokenless guest mints one and
returns it as `guestToken` on the cart (persist it and send it back on later calls).
Authenticated calls (Bearer) use the user's single active cart. Cart lines are keyed by
product **variant** (size + color); adding the same variant merges quantities.

| Method | Path | Body | Description |
| --- | --- | --- | --- |
| GET | `/cart` | – | Current cart (empty cart if none). |
| POST | `/cart/items` | `{variantId, quantity}` | Add / increment. Stock-checked. |
| PUT | `/cart/items/{variantId}` | `{quantity}` | Set absolute quantity (0 removes). |
| DELETE | `/cart/items/{variantId}` | – | Remove a line. |
| DELETE | `/cart` | – | Empty the cart. |

**Cart response**
```json
{
  "id","currency":"MXN",
  "items":[{"variantId","productId","slug","name","tech","garment","logo","colorHex",
            "size","sku","unitPriceMinor","unitPrice","quantity",
            "lineTotalMinor","lineTotal","available","inStock"}],
  "itemCount": 3, "subtotalMinor": 134700, "subtotal": "$1,347 MXN",
  "guestToken": "<only when a new guest cart is minted>"
}
```
Requesting more than available returns `409 conflict`.

## Orders — `/api/v1/orders`

| Method | Path | Auth | Body | Description |
| --- | --- | --- | --- | --- |
| POST | `/orders` | guest or Bearer | checkout | Convert the active cart into a `pending_payment` order. |
| GET | `/orders` | Bearer | – | The user's orders, newest first. |
| GET | `/orders/{id}` | Bearer | – | One order (owner, or any order for an admin). |

**Checkout body**
```json
{
  "email","name","phone?",
  "shippingAddress":{"recipientName","line1","line2?","neighborhood?","city","state",
                     "postalCode","countryCode"},
  "billingAddress?": { ...same... },
  "note?"
}
```
Checkout runs in **one transaction**: it locks inventory (`FOR UPDATE`), validates stock,
snapshots each line into `order_items`, **reserves stock** via the append-only
`inventory_movements` ledger, and marks the cart `converted`. Empty cart → `400`;
insufficient stock → `409`. Totals are in minor units; `shipping/tax/discount` are `0` in
v1.0.0 (`total = subtotal`).

## Admin — `/api/v1/admin` (admin bearer only)

Everyone is a `customer`; the single seeded `admin` manages the catalog. Every mutation
writes an `admin_audit_log` row. Non-admins get `403`.

| Method | Path | Body | Description |
| --- | --- | --- | --- |
| GET | `/admin/products` | – | All products (incl. draft/archived). |
| POST | `/admin/products` | ProductInput | Create; slug auto-derived & de-duped; builds variants + made-to-order inventory; optional collection link. |
| PUT | `/admin/products/{id}` | ProductInput | Update fields; re-syncs variants. |
| PATCH | `/admin/products/{id}/status` | `{status}` | `active` \| `draft` \| `archived`. |
| DELETE | `/admin/products/{id}` | – | Soft delete. |
| GET | `/admin/collections` | – | All collections + product counts. |
| POST | `/admin/collections` | `{name,description,sortOrder}` | Create. |
| PUT | `/admin/collections/{id}` | `{name,description,sortOrder}` | Update. |
| DELETE | `/admin/collections/{id}` | – | Soft delete. |

**ProductInput**
```json
{
  "name","description","garment","stack","tech","logoSlug",
  "price": 459, "colorHex": "#1a1a1a", "sizes": ["S","M","L"],
  "collectionSlug?": "new-arrivals", "featured": false, "status": "active"
}
```

## Payments — `/api/v1/payments`

Two providers: **Stripe** (cards) and **Coinbase Commerce** (crypto: BTC / ETH / SOL).
Keys are read from the environment; test Stripe keys (`STRIPE_TEST_SECRET_KEY`) are
preferred over live, and live keys are refused unless `STRIPE_ALLOW_LIVE=true`.

| Method | Path | Auth | Description |
| --- | --- | --- | --- |
| GET | `/payments/config` | – | `{stripe:{enabled,publishableKey}, crypto:{enabled,coins}, currency}` for the checkout UI. |
| POST | `/payments/stripe/intent` | guest/Bearer | Create (or return) a PaymentIntent for `{orderId}`. |
| POST | `/payments/stripe/webhook` | signature | Stripe event sink (raw body, `Stripe-Signature` verified). |
| POST | `/payments/crypto/charge` | guest/Bearer | Create (or return) a Coinbase Commerce charge for `{orderId}`. |
| POST | `/payments/crypto/webhook` | signature | Coinbase event sink (`X-CC-Webhook-Signature` HMAC verified). |

**Idempotency (once-only charge).** Each order has one stable payment idempotency key
(`stripe_order_<id>` / `coinbase_order_<id>`), enforced by `unique (provider,
idempotency_key)` on `payments` **and** passed to Stripe as its own idempotency key. A
double-clicked pay button therefore returns the **same** PaymentIntent/charge — never a
second one (`reused: true` on repeat). Webhooks are processed **once** via the
`stripe_webhook_events` / `crypto_webhook_events` inboxes (insert-by-event-id).

**On success** (`payment_intent.succeeded` / `charge:confirmed|resolved`): the payment is
marked `succeeded`, the order transitions `pending_payment → paid` (guarded, one-time), and
each line's reserved inventory is converted to a sale in the append-only ledger.

Intent response: `{paymentId, paymentIntentId, clientSecret, publishableKey, amountMinor,
currency, status, reused}`. Charge response: `{paymentId, chargeCode, hostedUrl,
amountMinor, currency, status, reused, coins}`. When a provider is unconfigured its
create endpoint returns `503 payments_unavailable`.

## Testing

```bash
go test ./...                                   # unit tests (no DB needed)
DATABASE_URL=... go test ./internal/server/...  # + full storefront→checkout integration test
```
Unit tests cover money formatting, slugify, JWT/refresh/password, Coinbase HMAC
verification, and client-IP parsing. The integration test drives register → catalog → cart
→ checkout → orders against a real DB and self-cleans (skips when no DB is reachable).

## Deployment

`backend/Dockerfile` builds a static binary on `golang:1.26-alpine` and runs it on
`alpine:3.20` as a non-root user, listening on `:8000`. All config is via environment
variables.
